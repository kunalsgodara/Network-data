\section{Wikidata Editor Network}

\subsection{Introduction}

To analyse the structure of collaborative discussions on Wikidata, three datasets of different sizes were selected. Using datasets of different scales allows observation of how network properties and information diffusion behaviour change as the community grows.

The datasets used in this analysis are:

\textbf{Small Network: BOT\_REQUESTS.csv (263 KB)}: a relatively small dataset containing discussions related to automated bot requests. Although small, it contains sufficient interaction patterns to form a meaningful interaction network.

\textbf{Medium Network: ADMINISTRATORS.csv (1.8 MB)}: a moderately sized dataset capturing interactions in administrative discussions, representing governance and coordination activities within the Wikidata community.

\textbf{Large Network: USERS.csv (10.4 MB)}: a large dataset representing general user discussion activity across multiple Wikidata pages and threads.

Analysing networks built from these three datasets allows comparison of structural properties such as clustering, connectivity, and diffusion behaviour across different network scales. The following sections describe the construction of these networks, the computation of structural metrics, and diffusion simulations used to study information propagation within the editor communities.

\subsection{Network Construction}

\subsubsection{Representation of Nodes and Edges}
The social structure of Wikidata discussions is represented as a network graph. In this network, nodes correspond to Wikidata editors, where each node represents a unique user identified by the \texttt{username} column in the dataset. Edges represent social interaction between editors. An undirected edge is created between two users if they both participated in the same discussion thread on a Wikidata page. This modelling approach captures co-participation in discussions as a proxy for social interaction within the Wikidata community. The graph is constructed as an undirected network, since participation in the same thread implies a mutual relationship between users rather than a directional interaction. Edges also contain a weight attribute, which represents the frequency of interactions between two users. Each time a pair of users appears together in another discussion thread, the weight of the corresponding edge is incremented. As a result, stronger connections emerge between users who frequently interact across multiple discussions.

\subsubsection{Data Structures and Additional Information}
The network is implemented using the NetworkX library, which provides efficient data structures for representing complex graphs in Python. Specifically, the graph is constructed using the \texttt{nx.Graph()} class, which models an undirected network. Before constructing the network, the dataset is processed using Pandas DataFrames. The raw discussion data is cleaned and organized to identify unique discussion threads and participating users. Information that cannot be directly represented through network topology is stored as node or edge attributes. In this implementation, the primary attribute is the edge weight, which represents the number of shared discussion threads between two users. NetworkX also allows additional metadata to be attached to nodes. If extended data such as geographic location, editor roles, or activity levels were available, these could be stored as node attributes using the \texttt{nx.set\_node\_attributes()} function.

\subsubsection{Algorithmic Approach}
The construction of the Wikidata editor network follows a structured pipeline. First, the CSV dataset is loaded into a Pandas DataFrame. A unique identifier for each discussion thread is then created by concatenating the \texttt{page\_name} and \texttt{thread\_subject} columns. This combined identifier ensures that each thread is uniquely represented in the dataset. Next, all unique usernames are extracted and added as nodes to the NetworkX graph. 

To generate connections between users, the dataset is grouped by the unique \texttt{thread\_id}. For each discussion thread, all participating users are identified. Using the \texttt{itertools.combinations} function, all possible pairs of users within the same thread are generated. Each pair represents a potential interaction between two editors. For every generated pair, an edge is added to the network graph. If the edge already exists, its weight attribute is incremented, indicating that the two users have interacted in multiple discussion threads. 

To avoid extremely dense subgraphs caused by very large discussions, an optional safety threshold (\texttt{max\_thread\_size}) can be applied. Threads exceeding this limit are excluded from pairwise edge generation, preventing a combinatorial explosion of edges and ensuring that the resulting network structure remains meaningful.

\subsubsection{Alternative Edge Definition}
To demonstrate the generality of the construction pipeline, we also extracted a directed variant of the small network. In this alternative definition, a directed edge is created from the first commenter (thread initiator) to all subsequent participants in each discussion thread. This captures asymmetric influence: who initiates discussions versus who responds. The resulting directed graph contains 527 nodes and 1,394 edges. The top-5 users by in-degree centrality are Pasleim (0.2624), Jura1 (0.1597), Mat\v{e}j Such\'{a}nek (0.1274), ValterVB (0.0798), and Ricordisamoa (0.0684). In contrast to the undirected network where degree centrality treats all connections equally, in-degree centrality in the directed variant specifically identifies editors who are frequently targeted as respondents, revealing a different dimension of influence within the community.

\subsection{Network Metrics}

\subsubsection{Characteristic Properties, Metrics, and Distributions}
To analyse the structural characteristics of the Wikidata editor network, we extracted the Largest Connected Component (LCC) for each of the three network scales (small, medium, and large). Focusing on the LCC ensures that all analysed nodes are reachable from each other, allowing meaningful computation of path-based metrics. 

The following structural metrics were computed for each LCC:
\textbf{Small Network CC:} Nodes = 519, Edges = 2,418, Density = 0.0179, Avg Clustering = 0.687, Avg Path Length = 2.60, Diameter = 6.
\textbf{Medium Network CC:} Nodes = 2,159, Edges = 11,912, Density = 0.0051, Avg Clustering = 0.647, Avg Path Length = 2.79, Diameter = 6.
\textbf{Large Network CC:} Nodes = 7,767, Edges = 24,219, Density = 0.0008, Avg Clustering = 0.244, Avg Path Length = 3.63, Diameter = 10.

As the network grows, density decreases significantly, which is typical for large social networks where the number of possible connections grows faster than the number of actual interactions. Despite this decrease in density, the networks maintain relatively high clustering coefficients and short average path lengths. 

To further analyse the network structure, the degree distribution was plotted on a log--log scale. The distribution exhibits a heavy-tailed pattern, meaning that most editors have only a few connections while a small number of users have very large numbers of interactions. This behaviour is consistent with the hub structure commonly observed in scale-free social networks\cite{topic2, topic3, barabasi1999}, where highly active users form central points of connectivity.

\subsubsection{Comparison with a Random Network}
To evaluate whether the observed structural properties arise from random processes, we generated equivalent Erd\H{o}s--R\'{e}nyi random graphs\cite{topic3} with the same number of nodes $N$ and connection probability $p$, ensuring a similar expected number of edges.

\textbf{Small Random:} Clustering = 0.0149, Path Length = 3.08.
\textbf{Medium Random:} Clustering = 0.0048, Path Length = 3.47.
\textbf{Large Random:} Clustering = 0.0009, Path Length = 5.08.

Comparing the empirical Wikidata networks with these random graphs reveals substantial differences. The clustering coefficients in the real networks are orders of magnitude higher than those in the corresponding random graphs (for example, 0.687 compared to 0.0149 in the small network). At the same time, the average path lengths remain comparable or slightly shorter than those of the random graphs. This combination of high clustering and short path lengths is a defining characteristic of small-world networks\cite{topic2, topic3, watts1998}. 

To quantify this property, we calculated the small-world coefficient:

$$S = \frac{C/C_{rand}}{L/L_{rand}}$$

where $C$ and $L$ represent the clustering coefficient and average path length of the empirical network, and $C_{rand}$ and $L_{rand}$ correspond to those of the equivalent random network. 

The resulting values were: \textbf{Small graph:} $S = 47.64$, \textbf{Medium graph:} $S = 157.34$, \textbf{Large graph:} $S = 439.66$. Since $S \gg 1$ in all cases, the small-world property is strongly supported\cite{watts1998}.

Beyond clustering and path length, the Wikidata networks also diverge from random graphs in their degree distribution shape and transitivity. The global transitivity (ratio of closed triangles to connected triples) was computed for each network and its random counterpart: Small network transitivity was 0.1745 (random: 0.0161), Medium was 0.1390 (random: 0.0049), and Large was 0.0777 (random: 0.0009). In all cases, the real networks exhibit transitivity values that are one to two orders of magnitude higher than the random baselines, confirming the strong tendency for editors to form tightly-knit triadic clusters. Additionally, fitting a power-law model to the small network's degree distribution on a log--log scale yielded an exponent of approximately $-1.07$. In contrast, Erd\H{o}s--R\'{e}nyi random graphs follow a Poisson degree distribution, which lacks the heavy tail observed in the empirical data. This further confirms that the Wikidata networks exhibit scale-free characteristics not captured by simple random graph models.

\subsubsection{Representativeness of Editor Networks}
Viewing this network as a representation of collaborative knowledge production suggests that Wikidata discussions are driven by a core group of highly active editors who serve as hubs within the interaction network. These users participate in numerous discussion threads and connect otherwise separate groups of contributors. The high clustering coefficients indicate that editors tend to interact within overlapping conversational communities, often centred around shared topics or collaborative tasks. In such environments, if two editors both interact with a third editor, they are highly likely to interact with each other as well. 

Meanwhile, the relatively short average path lengths imply that information, opinions, and policy discussions can spread rapidly throughout the community. In practice, a message or idea typically needs to pass through only two to three conversational steps to reach most other users in the network. The presence of hub editors therefore helps maintain strong connectivity across the community while enabling efficient information diffusion across the broader Wikidata ecosystem.

The co-participation edge definition used in this analysis naturally inflates clustering coefficients because each discussion thread creates a clique among all its participants. A directed definition, such as the initiator-to-responder variant demonstrated above, would reveal asymmetric influence patterns that the undirected model cannot capture. Similarly, weighting edges by interaction frequency versus treating them as binary connections would alter betweenness centrality rankings, potentially highlighting different bridging users. The \texttt{max\_thread\_size} threshold also plays an important role: without it, large talk-page discussions with hundreds of participants would generate dense subgraphs that inflate hub connectivity artificially. These representation choices directly shape the structural properties observed and should be considered when interpreting network metrics.

\subsection{Epidemic Models}

\subsubsection{Judging the Plausibility of Non-Propagation}
To evaluate how plausible it is that a potentially controversial behaviour has not yet propagated through the network, it is necessary to examine the topological structure of the editor interaction graph. In particular, metrics such as shortest path length, degree centrality, and betweenness centrality help determine how easily behaviours or information could spread between editors. If two suspicious editors are located close to each other in the network (for example, having a short shortest-path distance or sharing highly connected neighbors), then a highly contagious behaviour would be expected to propagate quickly across the network. 

Our epidemic spread simulations on the Wikidata networks (small, medium, and large) demonstrate the rapid nature of such cascades. In the large network, a diffusion process starting from a random node initially spreads slowly but then accelerates rapidly after a few propagation steps. In contrast, when the diffusion begins from nodes with high degree centrality or high betweenness centrality, the spread grows extremely quickly, reaching a large portion of the network within only a few steps. 

These results suggest that if suspicious editors occupy central positions in the network, a truly contagious behaviour would likely spread quickly to their neighbors and beyond\cite{topic4}. Therefore, if no unusual behaviour is observed among their connected editors, it is plausible that the behaviour remains localized rather than forming a large-scale cascade.

\paragraph{Scenario 1: Both editors suspicious.}
When both flagged editors are high-degree hub users, the situation is most critical. Using the small network as a worked example, we selected the two highest-degree nodes: \emph{Pasleim} (degree 226) and \emph{Jura1} (degree 198). These two editors are directly connected (shortest path = 1 hop) and share 105 common neighbours. Given their extreme proximity and extensive overlap, a contagious behaviour originating from either editor would almost certainly have propagated to the other and to the majority of their shared neighbourhood. The priority check list for investigation should therefore include the union of both editors' neighbourhoods, ranked by betweenness centrality. The top candidates are: Mat\v{e}j Such\'{a}nek (betweenness 0.1606), ValterVB (0.0652), Edoderoo (0.0577), Pigsonthewing (0.0455), and Multichill (0.0394). These users act as bridges to other communities and should be checked first.

\paragraph{Scenario 2: Only one editor suspicious.}
When only one editor is flagged and a second editor is randomly selected (representing a low-risk user), the situation differs substantially. Using \emph{Pasleim} (degree 226) as the suspicious editor and \emph{GualdimG} (degree 1) as the random editor, the shortest path is 2 hops with only 1 common neighbour. In this case, propagation is far less certain. The priority check list focuses on the neighbours of the suspicious editor only, again ranked by betweenness centrality: Jura1 (0.2426), Mat\v{e}j Such\'{a}nek (0.1606), ValterVB (0.0652), Edoderoo (0.0577), and Pigsonthewing (0.0455).

\paragraph{Propagation Dynamics across Medium and Large Networks.}
While the small network demonstrates rapid contagion due to its ultra-dense core, the \textbf{medium} and \textbf{large} networks exhibit different propagation dynamics. As the network size increases (scaling to 2,159 and 7,767 nodes respectively), the overall density drops dramatically (to 0.0051 and 0.0008). In the medium network, while local clustering remains high, a contagious behaviour originating from an average user will take several more hops to reach the entire community. In the large network, the lower transitivity (0.0777) and lower average clustering coefficient indicate that many users exist on the distant periphery of the core discussion space. Consequently, in the sparsest (large) network, a single anomalous editor lacking hub connections is highly unlikely to trigger a rapid community-wide cascade. However, if two prominent administrators acting as hubs were compromised in the large network, the priority list of nodes to check must expand aggressively to account for the massively increased number of bridging connections they possess, leveraging betweenness centrality to staunch the diffusion across the sparser periphery.

\subsubsection{Priority List for Checking Editors}
To mitigate the risk of a potential behavioural cascade, the Wikimedia Foundation could use network-based prioritization when investigating editors connected to suspicious accounts. A practical strategy is to examine the immediate neighbors of the suspicious editors, ranking them according to their degree centrality and betweenness centrality\cite{topic3, topic4, newman2003}. Our simulations provide empirical support for this approach. Diffusion processes initiated from nodes with high betweenness centrality consistently spread to the largest fraction of the network within the smallest number of steps. Nodes with high betweenness act as bridges between otherwise separate communities, meaning that behaviours originating from such nodes can quickly propagate between different discussion groups. 

Consequently, monitoring editors with high centrality values, particularly those connected to suspicious users, can significantly improve the chances of detecting or containing potential cascades before they reach a large portion of the community.

\subsubsection{Comparative Summary and Network Characterisation}
Comparing the structural characterisation of our three results, we can situate these empirical interaction graphs along the \textbf{regular / small-world / random network continuum}. At one extreme, highly regular lattices have high clustering but extremely long path lengths; at the other extreme, Erd\H{o}s--R\'{e}nyi random networks exhibit short path lengths but critically lack clustering ($<0.01$ in our random baselines). All three Wikidata editor networks (Small, Medium, and Large) position themselves firmly in the \textbf{small-world} regime. They blend the short average path lengths of random graphs (ranging narrowly from 2.60 to 3.63 hops regardless of size) with the massively elevated clustering coefficients typical of regular social circles (0.687 down to 0.244). Furthermore, their heavy-tailed degree distributions distance them fundamentally from both regular and random architectures, demonstrating clear \emph{scale-free} features driven by dominant hub editors.

The specific choices made for \textbf{representing these networks profoundly impacted these results}. By choosing an undirected, co-participation model for bounding edges (where any two users in a thread are fully connected), we inherently inflate clustering coefficients since every thread forms an artificially complete clique. In contrast, if we had employed the alternative directed representation (linking only the initiator to responders), the resulting metrics would shift, suppressing the exaggerated triad transitivity and prioritizing asymmetric information influence rather than mutual social engagement. Likewise, our implementation of the \texttt{max\_thread\_size} cap directly shapes the final connectivity; without this choice, immense generic talk pages would act as synthetic super-hubs, completely collapsing the average path length and dominating betweenness centrality at the expense of capturing true, meaningful peer-to-peer discussions.

\section{PART 2: Leeds Road Network}

\subsection{Spatial Networks and Planarity}

\subsubsection{Definition of Study Area}
For Part 2, the driveable road network surrounding a dense cluster of accidents in Leeds city centre was extracted using the OSMnx library. The network was centred at the coordinates Latitude 53.7969, Longitude $-1.5460$ (approximately Easting 430000, Northing 433500). To satisfy the requirement of at least 300 accidents in the study area, we tested several buffer radii around this centre point: 500\,m yielded 138 accidents, 750\,m yielded 259 accidents, and 1000\,m yielded 403 accidents. A radius of 1000\,m was therefore selected, providing 403 recorded accidents within the study area and an approximate coverage of $\pi \times 1^2 \approx 3.14$\,km$^2$.

Using OSMnx, a street network graph was generated with a 1000-metre radius (\texttt{dist = 1000}), capturing the dense urban road structure within the city centre.

\subsubsection{Network Size and Density Metrics}
The extracted road network consists of the following structural properties:
\textbf{Number of Nodes:} 647 intersections or dead-end nodes, \textbf{Number of Edges:} 1,192 street segments.

From these values, spatial density measures were calculated:
\textbf{Node Density:} 205.9 nodes per km$^2$, \textbf{Edge Density:} 379.4 edges per km$^2$, \textbf{Average Street Length:} 72.1\,m, \textbf{Intersection Density:} 151.5 intersections per km$^2$ (counting nodes with degree $\geq 3$).

These relatively high density values are typical of a dense urban road network, particularly within historic UK city centres where street layouts evolved organically over long periods. The average street length of 72.1\,m reflects the short, irregular segments characteristic of older city centres. The intersection density of 151.5 per km$^2$ indicates a finely connected grid, supporting multiple route choices for drivers and pedestrians.

\subsubsection{Spatial Diameter and Circuitry}
To examine the connectivity and efficiency of the road network, two additional structural metrics were calculated.

\textbf{Spatial Diameter:} 44. The spatial diameter represents the longest shortest path between any pair of nodes in the network, measured in number of street segments. A value of 44 indicates that the most distant pair of nodes within the network requires traversing 44 edges when following the shortest available route.

\textbf{Average Circuitry:} 2.123. Circuitry\cite{topic6} measures the ratio between the actual shortest-path distance along the road network and the straight-line (Euclidean) distance between two points. A value of approximately 2.12 indicates that, on average, travellers must move more than twice the direct Euclidean distance when navigating the street network. This relatively high circuitry reflects the indirect and irregular street structure typical of older city centres, as well as the presence of one-way streets, pedestrianised zones, and constrained urban layouts.

\subsubsection{Planarity Test}
The planarity of the road network was tested using the \texttt{nx.check\_planarity()} function from NetworkX. 

\textbf{Result:} The network is non-planar. 

Although road networks are often conceptualised as planar graphs\cite{topic5}, where edges intersect only at nodes, real-world urban road systems rarely satisfy strict planarity. In modern cities, infrastructure such as overpasses, underpasses, bridges, and grade-separated interchanges create situations where road segments cross without forming an intersection node. In Leeds, structures such as sections of the Inner Ring Road introduce these types of crossings, causing the extracted road network to be classified as non-planar.

Figure~\ref{fig:planarity} shows a concrete example of a non-planar crossing in the Leeds road network: two road segments intersect geometrically (e.g.\ at an overpass/underpass) without sharing a node, violating the planarity condition.

\subsection{Road Accidents}

\subsubsection{Accident Distribution and Density}
To analyse the spatial distribution of traffic accidents within the study area in Leeds city centre, a regular grid of 100\,m $\times$ 100\,m cells was overlaid on the study area. Using a spatial join, accident records were aggregated within each grid cell.

The resulting density map reveals that the 403 recorded accidents are not uniformly distributed across the study area. These hotspots typically occur near major intersections or short segments of highly trafficked streets, suggesting that specific road configurations and traffic conditions strongly influence accident occurrence. Such clustering patterns are commonly observed in urban environments where vehicle flows, pedestrian activity, and signalised junctions interact.

\subsubsection{Spatial Clustering: K-function and Moran's I}
To evaluate whether accidents display statistically significant spatial clustering, two complementary analyses were performed.

First, Ripley's K-function\cite{ripley1976} was computed to test for clustering at multiple spatial scales. The K-function compares the observed number of accident pairs within distance $d$ against the expected count under Complete Spatial Randomness (CSR). As shown in Figure~\ref{fig:kfunction}, the observed K(d) lies above the theoretical CSR line at all measured distances, indicating that accidents are significantly clustered rather than randomly distributed across the study area.

The resulting Moran's I value of approximately 0.1128 (p-value = 0.001) indicates a statistically significant positive spatial autocorrelation. \subsection{Voronoi Diagram}

To plan four simultaneous marathon start points across Leeds, a Voronoi diagram\cite{topic5, okabe2000} was generated to partition the city into distinct catchment zones corresponding to each start location. Voronoi diagrams divide a geographic area such that every location within a zone is closest to its corresponding seed point compared to all other points. In the context of marathon planning, this approach ensures that participants are assigned to the nearest available start location, thereby minimizing travel distance and improving accessibility. 

Four representative start points were selected to provide broad spatial coverage of the city: Leeds city centre, west Leeds, east Leeds, and south Leeds. Using these points as seeds, Voronoi polygons were generated using Shapely's \texttt{voronoi\_diagram} function. The resulting polygons were then clipped to the official administrative boundary of Leeds, obtained through OSMnx, ensuring that the final zones correspond only to the city's geographic extent.

\textbf{Voronoi Type Rationale (Q2):} For this specific logistical routing application, a \textbf{node network Voronoi diagram} is fundamentally the most useful type (modelled here by spatially bounding and snapping the street intersections to the nearest geometric seed polygon). While a purely \emph{edge planar Voronoi} ignores the topology of the road network entirely (crossing un-navigable obstacles), and an \emph{edge points network Voronoi} over-complicates calculations by splitting individual road segments midway, the node network Voronoi optimally assigns contiguous valid road intersections to their closest macro-zone. This ensures that the generated 42-kilometre cyclic marathon paths solely route through unbroken, driveable nodes cleanly sequestered within one start zone's domain.

\subsubsection{Calculated Marathon Zone Areas}
The area of each clipped Voronoi polygon was calculated and converted into square kilometres, producing the following catchment sizes:
\textbf{Zone 0 (City Centre):} 100.73 km$^2$, \textbf{Zone 1 (West Leeds):} 17.53 km$^2$, \textbf{Zone 2 (East Leeds):} 96.40 km$^2$, \textbf{Zone 3 (South Leeds):} 109.46 km$^2$.

The results reveal a substantial variation in the size of the catchment zones. In particular, Zone 1 (West Leeds) is significantly smaller than the other regions. This likely occurs because the west Leeds start point is positioned relatively close to the other selected locations and is partially constrained by the city boundary, resulting in a smaller Voronoi region. Conversely, the south Leeds start point produces the largest zone, covering over 109 km$^2$.

\subsubsection{Marathon Route Finding (Q3)}
To identify marathon-length circular routes within each Voronoi zone, the full Leeds driveable road network was downloaded (31,585 nodes, 70,781 edges). Each of the four seed points was snapped to the nearest network node, and a randomised walk-based heuristic was used to find routes of approximately 42\,km that start and end at the seed node. The algorithm performs multiple random walks from the start node, and when the accumulated distance approaches the target, it computes the shortest path back to the start. Routes within a tolerance of $\pm 3$\,km are accepted.
